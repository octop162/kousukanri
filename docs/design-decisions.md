# 設計判断の記録

## 1. 縦軸タイムライン (Phase 1)

**決定**: 時間軸を縦方向 (上→下 = 朝→夜) にした。

**理由**: Toggl Track のカレンダービューと同じ直感的なマッピング。QGraphicsScene の座標系と自然に一致する。

**実装**: `PIXELS_PER_HOUR = 200` で1時間あたり200pxを確保。24時間分 = 4800px の縦長シーン。

## 2. 5分スナップ (Phase 1)

**決定**: すべての時刻操作を5分単位にスナップ。

**理由**: ピクセル精度の時刻は実用的でない。5分単位が細かすぎず粗すぎないバランス。

**実装**: `y_to_time()` 内で `round(total_minutes / 5) * 5` で一括処理。個別のビューでスナップ処理を書かなくて済む。

## 3. 1レーン制約 (Phase 1)

**決定**: 並行タスク (複数レーン) はサポートしない。

**理由**: PoC段階で複雑さを抑える。1人の作業は通常1つのタスクに集中するため。

**影響**: 重複チェック (`resolve_overlap`, `resolve_move`) が単純な1次元の区間問題になる。

## 4. シグナル駆動 + Controller 仲介 (Phase 1)

**決定**: Scene が Signal を発火し、Controller が受信してデータ更新 + 他ビューへ反映。

**理由**: ビュー間の直接参照を避け、疎結合を維持。DB層を後から差し込みやすい。

**パターン**:
```
Scene → task_created Signal → Controller._on_task_created()
  → self._tasks[id] = task
  → self._list_view.add_task(task)
```

## 5. インメモリ dict (Phase 1〜1.8)

**決定**: タスク/プロジェクトを `dict[str, Task/Project]` でメモリ上に保持。DB は Phase 2 で追加予定。

**理由**: まず UI の動作確認を優先。永続化は後回しにして開発速度を上げた。

**リスク**: アプリ終了時にデータが消える (現状は PoC なので許容)。

## 6. QSplitter レイアウト (Phase 1.5)

**決定**: 左にタイムライン、右にサイドパネル (タスクリスト等) を QSplitter で分割。

**理由**: ユーザーが好みに応じて幅を調整できる。Toggl Track のような2カラムレイアウト。

## 7. タブ化 (Phase 1.6)

**決定**: サイドパネルを QTabWidget で「タスク」「プロジェクト」「設定」に分割。

**理由**: 右側の縦スペースを有効活用。機能追加時にタブを増やすだけで済む。

## 8. プロジェクト → タスク色連動 (Phase 1.7)

**決定**: タスクにプロジェクトを紐付けると、プロジェクトの色がタスクブロックに反映される。プロジェクト削除時はタスクの色は維持し、project_id を None にする。

**理由**: Toggl Track と同じ挙動。色でプロジェクトを視覚的に区別できる。

## 9. Controller を QObject 継承に変更 (Phase 1.8)

**決定**: TaskController を素のクラスから QObject 継承に変更。

**理由**: QTimer を使ってタイマーの毎秒 tick を駆動するため。QObject でないと QTimer の parent にできない。

**影響**: 既存のシグナル接続コードには変更なし (connect は Scene 側の Signal を使っているため)。

## 10. タイマー中は重複チェックしない (Phase 1.8)

**決定**: タイマーで計測中のタスクは重複チェック (`resolve_overlap`) を行わない。

**理由**: 計測中のブロックは現在時刻から伸び続けるため、既存ブロックとの重複を動的に解決するのは複雑すぎる。PoC ではシンプルさを優先。

## 11. タイマー停止時の5分スナップ (Phase 1.8)

**決定**: ■ (停止) 押下時に end_time を5分単位にスナップして確定。

**理由**: 他のタスク作成と同じ粒度に揃える。12:03 に止めたら 12:05 になるなど、きりの良い時刻になる。
